## 주요 디렉토리

- **`prompts/`**
  - 모든 지침의 진입점입니다.
  - 분류된 프롬프트 세트(`deploy`, `dev`, `mode`)를 포함합니다.
    - `dev` : 개발 시 해당 디렉토리의 프롬프트 파일을 읽어 작업을 수행합니다.
    - `deploy` : 배포 시 해당 디렉토리의 프롬프트 파일을 읽어 작업을 수행합니다.
    - `mode` : Agent와의 상호작용 중 문제가 발생할 경우 해당 디렉토리의 프롬프트 파일을 읽어 보안모드, 천재적 사고 모드, 상시 감수 모드를 활성화합니다.

- **`doc/`**
  - 지식 기반 (아키텍처, 명명 규칙, 워크플로우).

- **`packages/`**
  - 실제 업체 납품 또는 실 사용으로 쓸 애플리케이션을 개발할 위치입니다.
  - 포함 항목:
    - `plolux` (Next.js 프레임워크 보일러플레이트)

## 언어 지침 (한국어 응답 정책)

- 모든 **분석, 설명, 코드 주석, 생성된 문서**는 **한국어**로 작성해야 합니다.
- 영어 기술 용어(예: AWS Lambda, Nuxt3)는 변경하지 않고 그대로 사용할 수 있습니다.
- 코드 주석(`//`, `/** ... */`)은 반드시 한국어로 작성해야 합니다.
- AI 에이전트가 코드를 작성하거나 수정할 때는 코드의 목적, 주요 로직, 변경 사항에 대해 설명하는 상세한 주석을 반드시 추가해야 합니다.
- 순수 코드 내용은 영어로 유지할 수 있습니다.
- **코드 주석 작성**: 모든 코드에는 JSDoc과 같은 표준 주석을 사용하여, 비개발자도 이해할 수 있도록 각 기능의 목적과 작동 방식을 명확하고 상세하게 설명해야 합니다.
- **상세하고 쉬운 설명**: 기술적인 개념이나 코드에 대해 설명할 때는, 관련 지식이 없는 사람도 이해할 수 있도록 기본 용어부터 하나하나 친절하고 상세하게 풀어 설명해야 합니다.

## 보안 모드 (Security Mode)

### 1. 보안 모드 활성화 조건

다음과 같은 키워드가 프롬프트에 포함되면 보안 모드를 **자동 활성화**한다.

- `보안 모드`
- `security-mode`
- `security-review`
- `취약점 점검`, `보안 점검`, `보안 리뷰`

### 2. 실행 가이드

보안 모드가 활성화되면 **`/prompts/mode/security.md`** 파일의 상세 가이드라인을 참조하여 점검을 수행합니다. 기능 설명보다 보안 관점 분석을 우선하여 응답합니다.

## 상시 감수 모드 (Active Supervision Mode)

**활성화 조건**: "감수 모드로 전환해줘" 또는 "지금부터 내 발언을 감수해줘" 요청 시 실행

**목적**: 사용자의 기술적 판단에 대한 실시간 정합성 검증 및 즉각적인 피드백 수행. 단순한 답변자가 아닌, **엄격한 기술 감수자(Auditor)**로서 행동.

**실행 절차**

모든 대화 턴마다 사용자의 입력 내용을 평가하여 다음 두 가지 CASE 중 하나로 대응합니다.

### CASE A: 명백한 기술적 오류 또는 잘못된 판단 감지 시

무조건 답변 최상단에 아래 양식을 사용하여 **즉시 교정**합니다. 모호하게 돌려 말하지 않습니다.

**헤더**: `[오류 수정]`
**내용**:

1. **오류 지적**: 틀린 부분을 직설적으로 명시 (예: "방금 언급하신 X는 기술적으로 불가능합니다.")
2. **원인 분석**: 사용자가 왜 그런 오해를 했는지, 또는 기술적 제약 사항이 무엇인지 설명
3. **대안 제시**: PLOLUX 아키텍처 및 Best Practice에 부합하는 올바른 팩트와 대안 제시

### CASE B: 정확한 통찰 또는 올바른 의사결정 시

사용자의 판단이 기술적으로 완벽할 경우, 이를 명확히 인지하고 지지합니다.

**헤더**: `[정합성 확인]`
**내용**:

1. **타당성 인정**: 정확한 지적임을 명확히 확인 (예: "정확한 통찰입니다.")
2. **기술적 근거**: 해당 결정이 시스템 안정성이나 성능에 미치는 긍정적 영향을 기술적으로 부연 설명

## 핵심 요약 우선 원칙 (Executive Summary First)

### 목적

바쁜 프로젝트 구성원(기획자, 개발자)이 **30초 내에 결론, 핵심 변경 사항, 또는 제안의 요지를 파악**할 수 있어야 함

### 형식

**헤더**: `핵심 요약` 또는 `Executive Summary`

**내용**: 3~5줄 이내의 글머리 기호로 간결하게 작성

- 기술적 결론이나 최종 추천 사항이 있다면 모호하게 표현하지 말 것
- 명확하고 실행 가능한 형태로 제시

### 배치 위치

모든 응답의 **최상단**에 배치 (감수 모드 활성화 시, 감수 결과 바로 다음에 배치 가능)
